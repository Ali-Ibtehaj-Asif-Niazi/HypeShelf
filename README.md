# HypeShelf

> **"Collect and share the stuff you're hyped about."**

HypeShelf is a production-quality full-stack web application where users log in and publicly share their favourite movie recommendations on one clean, real-time shelf. Staff can highlight top picks with a curated **Staff Pick** badge.

## Features

### Public (no account required)
- Browse the 20 latest movie recommendations in a responsive card grid
- Staff Picks are visually highlighted with an amber badge
- See who added each recommendation and when
- Click through to the external link for each recommendation
- Sign-in prompt to add your own picks

### Authenticated users
- View **all** recommendations with real-time updates
- Filter recommendations by genre (instant client-side filtering)
- Add new recommendations (title, genre, link, blurb)
- Delete **their own** recommendations

### Admins
- Delete **any** recommendation
- Toggle the **Staff Pick** badge on any recommendation

## Quick Start

### Prerequisites

- Node.js 18+
- A [Clerk](https://clerk.com) account (free tier works)
- A [Convex](https://convex.dev) account (free tier works)

### 1. Clone and install

```bash
git clone <your-repo-url>
cd HypeShelf
npm install
```

### 2. Initialize Convex

```bash
npx convex dev
```

This will:
- Prompt you to log in to Convex (or create a free account)
- Create a new Convex project (or link an existing one)
- Generate `convex/_generated/` with typed API bindings
- Push your schema and functions to Convex Cloud
- Print your `NEXT_PUBLIC_CONVEX_URL`

**Keep this terminal running** — it watches your `convex/` folder for changes.

### 3. Configure Clerk

1. Go to [clerk.com](https://clerk.com) → create a new application
2. In **API Keys**, copy:
   - `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`
   - `CLERK_SECRET_KEY`
3. Go to **JWT Templates** → **New template** → select the **Convex** preset
   - Name it exactly `convex`
   - Copy the **Issuer** URL (e.g. `https://YOUR_INSTANCE.clerk.accounts.dev`)

### 4. Set environment variables

```bash
cp .env.local.example .env.local
```

Open `.env.local` and fill in all values:

```env
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...
CLERK_SECRET_KEY=sk_test_...
NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in
NEXT_PUBLIC_CLERK_SIGN_UP_URL=/sign-up
NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL=/recs
NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL=/recs
NEXT_PUBLIC_CONVEX_URL=https://YOUR_DEPLOYMENT.convex.cloud
CLERK_JWT_ISSUER_DOMAIN=https://YOUR_INSTANCE.clerk.accounts.dev
```

### 5. Set the Convex environment variable

In the **Convex Dashboard** → your project → **Settings** → **Environment Variables**, add:

```
CLERK_JWT_ISSUER_DOMAIN = https://YOUR_INSTANCE.clerk.accounts.dev
```

This is what `convex/auth.config.ts` reads at runtime to verify Clerk JWTs.

### 6. Run the app

In a second terminal (keep `npx convex dev` running in the first):

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

### 7. Promote a user to admin

1. Sign up via the app
2. Open **Convex Dashboard** → **Data** → `users` table
3. Find your user row → click the `role` field → change `"user"` to `"admin"`

## Roles & Permissions

HypeShelf implements two roles, stored in the Convex `users` table:

| Action | `user` | `admin` |
|---|---|---|
| View public recommendations | ✅ | ✅ |
| View all recommendations | ✅ (signed in) | ✅ |
| Create recommendation | ✅ | ✅ |
| Delete **own** recommendation | ✅ | ✅ |
| Delete **any** recommendation | ❌ | ✅ |
| Mark / unmark Staff Pick | ❌ | ✅ |

**All permission checks run server-side inside Convex mutations.** The client receives the role only to show/hide UI elements — it cannot grant itself elevated permissions.

New users always start as `"user"`. Promotion to `"admin"` is done via the Convex dashboard (or a future admin panel).

See [SECURITY.md](./SECURITY.md) for the complete RBAC implementation and threat model.

## Data Model

### `users` table

| Field | Type | Notes |
|---|---|---|
| `_id` | `Id<"users">` | Auto-generated Convex ID |
| `clerkId` | `string` | Clerk user subject, from JWT |
| `name` | `string` | Display name, synced from Clerk |
| `role` | `"admin" \| "user"` | RBAC role — never accepted from clients |

Index: `by_clerk_id` on `["clerkId"]`

### `recommendations` table

| Field | Type | Notes |
|---|---|---|
| `_id` | `Id<"recommendations">` | Auto-generated Convex ID |
| `title` | `string` | Max 120 characters |
| `genre` | `string` | Max 60 characters |
| `link` | `string` | Must be a valid http/https URL |
| `blurb` | `string` | Max 300 characters |
| `userId` | `Id<"users">` | Foreign key → `users._id` |
| `createdAt` | `number` | Unix timestamp (ms) |
| `staffPick` | `boolean` | Curated highlight, admin-only |

Indexes: `by_created_at`, `by_genre`, `by_user`

## API Reference (Convex)

All functions are in `convex/`. TypeScript types are auto-generated in `convex/_generated/`.

### Queries

#### `recommendations.getLatestRecommendations`
```ts
// Auth: Public
// Returns: 20 most recent recs with joined userName field
```

#### `recommendations.getAllRecommendations`
```ts
// Auth: Required (throws if unauthenticated)
// Returns: All recs ordered by createdAt desc, with userName
```

#### `recommendations.getRecommendationsByGenre`
```ts
// Auth: Required
// Args: { genre: string }
// Returns: Recs matching the genre, with userName
```

#### `users.getMe`
```ts
// Auth: Required
// Returns: Full users document (including _id and role) | null
```

---

### Mutations

#### `recommendations.createRecommendation`
```ts
// Auth: Required
// Args: { title: string, genre: string, link: string, blurb: string }
// Validates: char limits, http/https URL, non-empty fields
// Sets staffPick: false — client cannot override this
```

#### `recommendations.deleteRecommendation`
```ts
// Auth: Required
// Args: { id: Id<"recommendations"> }
// RBAC: user → own recs only | admin → any rec
```

#### `recommendations.markStaffPick`
```ts
// Auth: Required
// Args: { id: Id<"recommendations">, staffPick: boolean }
// RBAC: admin only — throws for user role
```

#### `users.upsertUser`
```ts
// Auth: Required
// Args: { name: string }
// Creates user with role: "user" on first call; updates name on repeat calls
// Idempotent — safe to call on every page mount
```

## Design Decisions

### Real-time with Convex
Convex's `useQuery` hook sets up a live WebSocket subscription. Any recommendation added, deleted, or updated by any user is instantly reflected on all connected clients — no polling required.

### Roles in Convex, not Clerk metadata
Clerk's `publicMetadata` is readable on the client and could be forged in Convex function calls. Storing roles in the Convex `users` table means permission checks happen entirely server-side, against data the client cannot tamper with.

### `useConvexAuth` guard on authenticated queries
After sign-in, `ConvexProviderWithClerk` needs a brief moment to forward the Clerk JWT to Convex. Passing `"skip"` to `useQuery` during this window prevents the query from firing before the auth handshake completes, avoiding the `"Not authenticated"` error on first load.

### `useEnsureUser` hook for user sync
Clerk owns the identity; Convex owns the role. The `useEnsureUser` hook calls `upsertUser` on every authenticated page mount to keep both in sync. It is idempotent and only updates the name if it has changed.

### Client-side genre filtering
For the scale HypeShelf targets, fetching all records and filtering locally over a real-time subscription is faster (no extra round-trip) and simpler than issuing a new server query on every filter change. `getRecommendationsByGenre` remains available for future server-side filtering or pagination.

## Security

See [SECURITY.md](./SECURITY.md) for the full threat model and RBAC documentation.

Quick summary:

| Threat | Mitigation |
|---|---|
| Unauthenticated write | `ctx.auth.getUserIdentity()` required at top of every mutation |
| Client-supplied role escalation | Role read from DB only, never from mutation args |
| Deleting others' content | `userId === rec.userId` ownership check before delete |
| Malicious URL injection | Server-side `new URL()` validation; `http/https` protocol enforced |
| XSS via link | `rel="noopener noreferrer"`; no `dangerouslySetInnerHTML` used |
| Oversized input | Server-side char limits: title ≤120, genre ≤60, blurb ≤300 |
| JWT forgery | Convex verifies signatures against Clerk's JWKS endpoint |
| Secret key leakage | `CLERK_SECRET_KEY` is server-only; never in `NEXT_PUBLIC_*` |

## Potential Improvements

| Feature | Description |
|---|---|
| **Pagination** | Convex cursor-based pagination for large datasets |
| **Full-text search** | Convex `searchIndex` on `title` + `blurb` |
| **Movie posters** | Fetch poster art from OMDB/TMDB via a Convex action |
| **Optimistic UI** | Instant card removal on delete using Convex optimistic updates |
| **Rate limiting** | Cap submissions per user per day with a `lastCreatedAt` check |
| **Email on Staff Pick** | Convex action → Resend/Postmark notification to the author |
| **Admin dashboard** | `/admin` page for user management without the Convex dashboard |
| **Image upload** | Custom posters via Convex file storage |
| **Vercel deployment** | `npx convex deploy` + Vercel for one-command production deploy |
